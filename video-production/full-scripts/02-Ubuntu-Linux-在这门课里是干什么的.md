## 正式脚本

大家好，这条视频我们来讲一个很多同学第一次上这门课时都会问的问题：

**为什么这门课要用 Ubuntu？或者说，为什么要用 Linux？**

这个问题其实特别正常。
因为大部分同学平时更熟悉的是 Windows，
平时写作业、上网、装软件、玩游戏，很多时候接触的也都是 Windows 的环境。

所以第一次看到老师在机器人项目里打开一个终端，
然后开始输入命令、切目录、激活环境，
很多同学心里会有一个很直接的反应：

“为什么非要搞得这么复杂？”

甚至会有人觉得，
是不是因为机器人项目本身就很难，
所以才故意要用一个看起来更‘工程化’的系统。

其实不是这样的。

这门课会用 Ubuntu，不是为了增加神秘感，
也不是为了故意把门槛拉高，
而是因为在真实的机器人、AI、嵌入式和开发环境里，Ubuntu 确实是一个非常常见、也非常实用的基础系统。

所以这条视频，我们就把这件事彻底讲清楚。

### 第一部分：Ubuntu 到底是什么

先别急着把它想得太复杂。

Ubuntu 本质上也是一个操作系统。

就像你平时用 Windows 一样，
Ubuntu 也是让电脑运行起来的一整套基础系统。

也就是说，
它不是一个单独的软件，
也不是某种“高级编程工具”，
它本身就是电脑工作的底层环境。

只不过和 Windows 相比，
Ubuntu 更常出现在下面这些场景里：
- 开发
- 服务器
- 人工智能
- 机器人
- 嵌入式设备
- 自动化系统

原因不是因为它“更酷”，
而是因为它在这些场景里确实更方便，也更稳定。

所以如果你是第一次接触 Ubuntu，
你先不要把它理解成“一个我必须全部研究透的系统”，
你可以先把它理解成：

**这是我们这门课整套项目运行和开发的工作台。**

这个理解就已经很够用了。

---

### 第二部分：为什么机器人项目特别喜欢用 Ubuntu

那接下来我们就说最核心的问题：
为什么机器人项目里，Ubuntu 会这么常见？

这里最主要有几个原因。

#### 第一个原因：很多开发工具天然更适合在这类环境里工作

机器人项目通常不会只碰一个软件。
你后面会慢慢接触到很多不同层级的东西，比如：
- Python
- 深度学习相关库
- 相机驱动
- 视觉处理工具
- 机械臂控制脚本
- 某些依赖管理工具
- 各种命令行调试工具

这些工具和库，有很多本来就是在 Linux 环境里被广泛使用和维护的。

换句话说，
在 Ubuntu 这类系统里，
它们更容易被装好，也更容易彼此兼容。

#### 第二个原因：它更适合“工程式工作”

你会发现，机器人项目和我们平时单纯用一个图形软件不太一样。

它经常需要你做这些事情：
- 运行命令
- 检查日志
- 安装依赖
- 管理环境
- 看设备状态
- 排查某个脚本为什么没有正常工作

这种工作方式，本来就更偏工程开发。

而 Ubuntu / Linux 在这种场景下，非常自然。
因为它本来就很适合：
- 命令行操作
- 脚本化流程
- 环境配置
- 服务和设备管理

这也是为什么很多真实的机器人开发环境，不会把重点放在“点鼠标”上，
而是更强调：

**你能不能真正控制项目运行的整个过程。**

#### 第三个原因：稳定和可复制性更强

对于课程项目来说，还有一个特别现实的原因。

那就是：
如果你希望一套项目可以被重复搭建、重复教学、重复验证，
那开发环境就必须尽量统一。

Ubuntu 在这方面有一个很大的优势，
就是它更容易形成一套标准流程。

比如：
- 统一系统版本
- 统一依赖安装方式
- 统一命令执行方式
- 统一环境管理方式

这对老师、助教和学生都很重要。

因为教学最怕的不是“项目本身难”，
而是每个人的机器都不一样、路径都不一样、装法都不一样，最后谁也说不清问题到底出在哪。

而 Ubuntu 的好处之一，就是更容易把这些流程固定下来。

### 第三部分：那这门课里，Ubuntu 到底拿来干什么

如果你从课程视角来理解，
Ubuntu 在这门课里并不是单独的学习目标，
而是整个项目的基础承载环境。

也就是说，后面很多你真正会做的事情，都会在这个环境里完成。

比如你会在 Ubuntu 里：
- 进入项目目录
- 激活 Conda 环境
- 运行自检脚本
- 调用相机脚本
- 跑目标检测
- 看深度图和 XYZ 输出
- 运行标定脚本
- 控制机械臂
- 检查设备和日志

你可以发现，
Ubuntu 在这里并不是一门单独存在的“理论课”，
而是所有后续动作发生的舞台。

如果没有这个舞台，
后面的很多脚本和工具其实都没地方稳定运行。

所以更准确地说，
我们不是在“为了学 Ubuntu 而学 Ubuntu”，
而是在：

**为了让机器人项目真正跑起来，需要在 Ubuntu 这个环境里工作。**

这个顺序一定不要理解反了。

---

### 第四部分：学生到底要学到什么程度

这里我也特别想替很多第一次接触的同学减轻一点心理压力。

因为一听到 Linux、终端、命令行，
很多人脑子里马上会出现一种印象：
“是不是我要把整个系统都学会？”

其实不是。

在当前阶段，你不需要一开始就把 Ubuntu 的所有机制都研究透。

你真正要掌握的，是下面这些更实际的能力：
- 会打开终端
- 会进入项目目录
- 会运行老师要求的命令
- 会看命令输出
- 会判断当前是成功还是失败
- 会在出问题时知道先看哪一层

也就是说，
你不是要先变成 Linux 专家，
才能开始学机器人。

而是在学机器人项目的过程中，
逐渐学会如何在这个开发环境里工作。

这是两个完全不同的感受。

前一种会让人觉得门槛特别高，
后一种就会合理很多。

所以你只要记住：

**当前阶段，你的目标不是精通 Ubuntu，而是在 Ubuntu 里把项目跑起来。**

这就已经很重要了。

### 第五部分：为什么这不是“额外负担”，而是机器人能力的一部分

很多同学在刚开始的时候，会有一个很真实的疑问：

“我明明是来学机器人，为什么还要学这些系统和命令行的东西？”

这个问题其实特别值得讲清楚。

因为现实里的机器人项目，真的不是“只会写一点控制代码”就够了。

一个真正的机器人系统，经常就是由这些层一起组成的：
- 系统环境
- 编程环境
- 驱动和设备连接
- 感知模块
- 决策模块
- 控制模块
- 调试和日志系统

也就是说，
机器人并不是一个脱离计算机环境独立存在的东西。

它背后本来就依赖一整套开发和运行环境。

所以你今天学会：
- 在 Ubuntu 里工作
- 在终端里执行标准流程
- 看懂基础输出
- 知道排错时先查哪里

这些能力不是绕路，
而是你在真正接近机器人项目本身。

换句话说，
这不是“额外增加的负担”，
而是机器人系统实践的一部分。

如果课程完全避开这些，
反而很容易让学生产生一种错觉：
好像机器人项目只是点几下按钮就会成功。

但真实的工程不是这样。

我们反而希望大家在课程里慢慢建立一种更真实的认识：

**机器人系统，是跑在环境里的，是要靠工程流程被搭起来的。**

---

### 第六部分：你现在最应该带着什么心态去学 Ubuntu

最后我想给学生一个比较轻松、但也比较正确的学习心态。

如果你现在刚开始接触 Ubuntu，
最好的心态不是：
“我要一下子把 Linux 全部学明白。”

而是：

“我要先学会在这个环境里完成这门课当前需要完成的事情。”

比如你现在只需要先做到：
- 能打开终端
- 能进入目录
- 能执行基础命令
- 能跑项目脚本
- 能看懂最基本的输出

这就已经是一个非常好的起点了。

后面随着项目继续推进，
你会慢慢越来越熟悉这个环境，
再逐渐理解更多东西。

这种学习方式其实比一开始硬背一堆系统知识更有效。

因为你是在真实使用里理解它，
而不是在抽象概念里死记它。
